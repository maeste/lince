# nono Integration Guide (experimental)

> **Status: experimental** — nono backend support is new and has not been extensively validated. See [../to-be-validated.md](../to-be-validated.md) for testing instructions.

[nono](https://github.com/always-further/nono) is an alternative sandbox backend for AI coding agents. It uses Landlock LSM on Linux and Seatbelt on macOS, providing kernel-enforced, capability-based isolation.

**macOS users**: agent-sandbox is Linux-only (it requires bubblewrap, which depends on Linux namespaces). nono is the supported sandbox solution on macOS.

## When to use nono

| Scenario | Recommendation |
|----------|---------------|
| Linux, want simple zero-dep sandbox | agent-sandbox (default) |
| Linux, want finer-grained per-path control | nono |
| Linux, kernel 5.13+ and want Landlock | nono |
| macOS | nono (only option) |
| Need credential proxy isolation | agent-sandbox (nono has built-in proxy too) |
| Need filesystem rollback | Both (agent-sandbox: snapshots, nono: `nono undo`) |

## Setup

### 1. Install nono

**macOS:**
```bash
brew install nono
```

**Linux:**
```bash
# Homebrew
brew install nono

# Cargo
cargo install nono-cli

# Or download prebuilt binary from GitHub releases
```

Verify:
```bash
nono --version
```

### 2. Generate profiles from lince config

```bash
agent-sandbox nono-sync
```

This reads your lince agent config (`agents-defaults.toml` + `config.toml`) and generates nono JSON profiles at `~/.config/nono/profiles/lince-<agent>.json`.

Options:
```bash
agent-sandbox nono-sync --dry-run          # print without writing
agent-sandbox nono-sync --agent claude     # sync one agent only
agent-sandbox nono-sync --agent codex -P default  # include profile env vars
```

### 3. Set backend

Edit `~/.agent-sandbox/config.toml`:

```toml
[sandbox]
backend = "nono"      # always use nono
# backend = "auto"    # prefer agent-sandbox on Linux, nono on macOS (default)
```

### 4. Verify

```bash
agent-sandbox status   # shows detected backends and active backend
nono run --profile lince-claude -- claude --version  # test nono profile
```

## Profile Mapping Reference

The `nono-sync` command translates lince config fields to nono JSON profile fields:

| lince config | nono profile | Notes |
|-------------|-------------|-------|
| `[agents.<name>].command` | CLI args after `--` | Not in profile (passed via `nono run -- <cmd>`) |
| `[agents.<name>].default_args` | CLI args after `--` | Appended to command |
| `[agents.<name>].env` + profile env | `environment.passthrough` | Env vars to pass through |
| `[env].passthrough` | `environment.passthrough` | Merged with agent env |
| `[sandbox].extra_rw` | `filesystem.write` | Writable directories |
| `[sandbox].extra_ro` + `ro_dirs` | `filesystem.read` | Read-only directories |
| `[sandbox].home_ro_dirs` | `filesystem.read` | Home subdirs, read-only |
| `[security].block_git_push` | `security.allowed_commands` | git excluded from allowed |
| `[security].credential_proxy` | N/A | nono has its own proxy approach |

### Generated profile structure

```json
{
  "extends": "claude-code",
  "meta": {
    "name": "lince-claude",
    "description": "Generated by lince agent-sandbox (nono-sync)"
  },
  "filesystem": {
    "allow": ["$WORKDIR"],
    "read": ["/home/user/.config/gcloud"]
  },
  "security": {
    "signal_mode": "allow_same_sandbox"
  }
}
```

### Built-in profile inheritance

When nono has a built-in profile for an agent, the generated profile uses `"extends"` to inherit from it:

| lince agent | nono built-in | Generated profile extends |
|-------------|--------------|--------------------------|
| `claude` | `claude-code` | `"extends": "claude-code"` |
| `codex` | `codex` | `"extends": "codex"` |
| `opencode` | `opencode` | `"extends": "opencode"` |
| `gemini` | *(none)* | Standalone profile |
| `aider` | *(none)* | Standalone profile |
| `amp` | *(none)* | Standalone profile |

## Features that don't translate

Some lince config features have no nono equivalent. `nono-sync` prints warnings for these:

| lince feature | nono equivalent | Warning |
|--------------|----------------|---------|
| `bwrap_conflict` | N/A | nono handles nested sandboxes differently |
| `disable_inner_sandbox_args` | N/A | Not needed with nono |
| `[snapshot]` settings | `nono undo` | nono has built-in rollback |
| `credential_proxy` | nono's own proxy | Different mechanism |
| PID namespace (`unshare_pid`) | Signal scoping (Landlock V6) | Partial equivalent |

## Feature comparison

| Feature | agent-sandbox | nono |
|---------|--------------|------|
| Isolation tech | bubblewrap (namespaces) | Landlock LSM / Seatbelt |
| Kernel requirement | 3.8+ | 5.13+ (full network: 6.6+) |
| Platform | Linux only | Linux + macOS |
| Per-path control | Mount-based (dir level) | Per-path read/write |
| Credential proxy | Built-in (opt-in) | Built-in |
| Filesystem rollback | Snapshots (rsync) | Content-addressable undo |
| Learn mode | strace-based | strace / fs_usage |
| Config diff/merge | Built-in | N/A |
| Dependencies | Zero (Python stdlib) | Rust binary + deps |
| PID isolation | Full namespace | Signal scoping (V6) |

See `docs/comparison-agent-sandbox-vs-nono.md` for the full analysis.

## Troubleshooting

### "nono: command not found"

Install nono. On macOS: `brew install nono`. On Linux: `cargo install nono-cli` or download from GitHub releases.

### "Profile lince-claude not found"

Run `agent-sandbox nono-sync` to generate profiles. Verify with `ls ~/.config/nono/profiles/lince-*.json`.

### "Landlock not supported"

Your kernel is too old. Landlock requires Linux 5.13+. Check with `uname -r`. Consider using agent-sandbox instead.

### agent-sandbox commands on macOS

Most agent-sandbox commands (init, diff, merge, snapshot, learn) require bubblewrap and won't work on macOS. Use nono's built-in equivalents:
- `nono undo` instead of `agent-sandbox snapshot-restore`
- `nono learn` instead of `agent-sandbox learn`
- `nono why <path>` instead of `agent-sandbox status`

The `nono-sync` and `status` commands work on all platforms since they don't use bubblewrap.
